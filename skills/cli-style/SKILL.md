# CLI Style

适用于仓库内脚本或命令行工具的新增与重构，目标是统一 CLI 结构、帮助文案、补全风格和最小自测。

## 何时使用

- 新增一个命令行脚本。
- 把旧的 flag 驱动接口重构为更清晰的命令结构。
- 调整 `--help`、子命令帮助或参数校验方式。
- 编写或修改 `zsh/completions/` 下的补全文件。
- 需要验证 CLI 的正常路径和常见错误路径。

## 交互模型

- 顶层优先使用“命令名 + 子命令 + 子命令选项/参数”的结构。
- 真正执行动作的入口优先做成子命令，不要把动作继续堆成顶层长选项。
- 全局选项只保留元信息类入口，优先支持 `--help` 和 `-h`。
- 子命令自身也应支持局部 `--help` 和 `-h`。
- 位置参数保持稳定和精简，只有在确实需要可选修饰时才加子命令级选项。

## Help 约定

- 顶层 `--help` 输出全局 usage、子命令列表和简短说明。
- `help` 子命令可以保留，作为 `--help` 的可发现入口。
- 子命令 `--help` 只展示该子命令的 usage 和局部选项，不回退成整页全局帮助。
- 错误信息先指出问题，再附对应层级的 help。
- 缺参、未知参数、未知子命令都要返回非零退出码。

## Completion 约定

- 顶层补全优先采用 `git` 风格的“候选项 + 简短说明”。
- 子命令、选项和资源名分层补全，不要把所有情况混在一个平面里。
- 资源名补全优先展示真实候选值；如果没有稳定元数据，不必强行补描述。
- 补全文件放在 `zsh/completions/`，命名与命令一致。
- 调整 CLI 时，补全必须同步更新。

## 仓库联动

- 修改 CLI 入口时，同步检查仓库内是否有旧调用残留。
- 如果 shell 启动脚本或其他脚本调用了该命令，一并改成新接口。
- 不要因为局部 CLI 微调就修改根 `AGENTS.md` 的实现规则；只有新增 skills 入口或仓库级导航事实时才更新根文档。

## 最小自测

- 语法检查：脚本本身和相关 completion 文件至少做一次 shell `-n` 检查。
- 帮助检查：验证顶层 `--help` 和至少一个子命令 `--help`。
- 正常路径：至少覆盖一条完整主流程。
- 错误路径：至少覆盖未知子命令、缺参或非法参数中的一类。
- 如果命令涉及用户真实配置、密钥或仓库数据，优先用临时目录和临时环境变量做隔离测试。

## 实施顺序

1. 先确认现有 CLI 入口、补全文件和仓库内调用点。
2. 收敛命令结构，再改 help 和错误信息。
3. 同步更新 completion。
4. 搜索并改掉仓库里的旧调用。
5. 做最小自测，记录实际验证范围和未覆盖项。
